# คู่มืออ่านกราฟ Analytics — สำหรับคนที่ไม่มีพื้นสถิติ

> เปิดหน้านี้ได้จากแอปโดยกด **"ดูสรุปภาพรวม"** หลังถ่ายรูป หรือจากเมนูหน้าแรก
> กราฟทุกตัวอัปเดตแบบ realtime ตาม threshold slider และ filter ด้านบน

---

## ภาพรวม: หน้านี้คืออะไร?

หน้า Analytics ช่วยตอบ 2 คำถามหลัก:

1. **ควรตั้ง threshold ที่เท่าไหร่?** — ยิ่งสูง ยิ่งเข้มงวด ยิ่งปลอดภัย แต่คนจริงอาจโดน reject มากขึ้น
2. **ระบบทำงานดีแค่ไหน?** — แยก Live vs Spoof ได้ชัดแค่ไหน มีเคสที่ผิดกี่เปอร์เซ็นต์

### ศัพท์พื้นฐานก่อนอ่านกราฟ

| คำ | ความหมายสั้นๆ |
|---|---|
| **face_score** | คะแนน 0–100% ที่ AI ให้กับรูปว่า "มั่นใจแค่ไหนว่าเป็นคนจริง" |
| **threshold** | เส้นตัดสิน: score ≥ threshold → ผ่าน, < threshold → ไม่ผ่าน |
| **Live** | เคสที่เราตั้งไว้ว่าเป็นคนจริง (ตั้งใน editor ล่างสุดของหน้า) |
| **Spoof** | เคสที่เราตั้งไว้ว่าเป็นการหลอก (รูปถ่าย, วิดีโอ, หน้ากาก) |

---

## Filter & Threshold Slider

**Filter** ด้านบนสุด — เลือก date range, test_case ที่ต้องการดู, และเคสที่ถูก reject จาก check อื่น (เช่น hand detected) เพื่อให้กราฟโฟกัสเฉพาะเคสที่เกี่ยวข้อง

**Threshold slider** — ลากซ้ายขวา แล้วดูกราฟทุกตัวอัปเดตทันที ใช้สำรวจว่า "ถ้าตั้ง threshold ที่ X% กราฟเปลี่ยนยังไง"

---

## KPI 4 ตัว (แถวตัวเลขบนสุด)

| ตัว | ความหมาย |
|---|---|
| **ทั้งหมด** | จำนวน attempt ในช่วงเวลาที่เลือก |
| **mean (μ)** | ค่าเฉลี่ย: บวก score ทุกเคสแล้วหารจำนวน |
| **median** | ค่ากลาง: เรียง score ทุกเคส แล้วเอาตัวที่อยู่ตรงกลาง (ไม่ถูกค่าสุดโต่งดึง) |
| **pass** | % ของเคสที่ score ≥ threshold ปัจจุบัน |

> **mean กับ median ต่างกันยังไง?** ถ้ามีเคส score 2% อยู่ 1 เคส (outlier) mean จะถูกดึงลงต่ำ แต่ median ไม่กระทบ — median จึงบอก "คนทั่วไปได้เท่าไหร่" ได้ตรงกว่า

---

## กราฟที่ 1 — Pass-rate vs Threshold

### ดูยังไง
- **แกน X** (ซ้าย → ขวา): threshold 0–100%
- **แกน Y** (ล่าง → บน): % ของเคสที่จะผ่าน
- **เส้นขาว**: กราฟ curve ลาดลงจากบนซ้ายไปล่างขวา — threshold สูงขึ้น ผ่านน้อยลง
- **เส้นฟ้าเข้ม (จุดๆ)**: threshold ที่เลือกอยู่ตอนนี้ พร้อมป้ายบอก "T → pass XX%"

### เอาไปทำอะไรต่อ

ตอบคำถาม **"ถ้าตั้ง threshold ที่ X% จะมีกี่ % ของเคสที่ผ่าน?"**

ตัวอย่าง: ถ้าเส้นฟ้าอยู่ที่ threshold 70% แล้วป้ายบอก "pass 85%" หมายความว่า 85 ใน 100 เคสจะผ่านระบบ

ใช้กราฟนี้เพื่อ **ประมาณผลกระทบก่อนเปลี่ยน threshold จริง** เช่น "ถ้าขึ้น threshold อีก 10% จะเสีย UX กี่ %?"

## กราฟที่ 2 — Bell Curve (การกระจายของ face_score)

### ดูยังไง
- รูปทรงระฆังคว่ำ — ยอดระฆัง = score ที่เกิดขึ้นบ่อยที่สุด, ฐานกว้าง = score กระจายออกไปสองข้าง
- **แกน X**: score 0–100%
- **แกน Y**: ความหนาแน่น (พื้นที่ใต้ระฆัง = 100% ของทุกเคส)

| เส้น | สี | ความหมาย |
|---|---|---|
| T | ฟ้าเข้ม (จุดๆ) | threshold ปัจจุบัน |
| μ | เหลือง | mean (ค่าเฉลี่ย) |
| Med | ส้ม | median (ค่ากลาง) |
| เคสนี้ | ชมพูสด | score ของการถ่ายรูปครั้งนี้ (มีเฉพาะเมื่อเปิดจากหน้า Result) |

**Gesture**: pinch zoom เพื่อขยายดูบริเวณที่สนใจ, double-tap เพื่อ reset กลับมุมมองปกติ

**Toggle Count / Density** (มุมขวาบน): ปรับสเกล Y — รูปร่างระฆังเหมือนกัน แต่ตัวเลขแกน Y ต่างกัน

### เอาไปทำอะไรต่อ

- **เคสนี้อยู่จุดไหน?** — ถ้าเส้นชมพูอยู่ทางขวาของ threshold (ฟ้า) = ผ่าน, ซ้าย = ไม่ผ่าน
- **ระฆังแคบ vs กว้าง**: ระฆังแคบ = score ของทุกคนคล้ายกัน (ระบบ consistent), ระฆังกว้าง = score กระจาย (บางคนได้สูงมาก บางคนได้ต่ำมาก)
- **threshold อยู่ไกลยอดระฆังแค่ไหน**: ยิ่ง threshold ห่างจากยอดไปทางซ้าย ยิ่งมีส่วนใหญ่ผ่าน

> **ระวัง**: ระฆังนี้คือ "โมเดลทางทฤษฎี (Gaussian fit)" — ระบบวาดระฆังที่เข้ากับค่าเฉลี่ยและ spread ของข้อมูล **ไม่ใช่** bar chart ของข้อมูลจริง ถ้าอยากดูข้อมูลดิบดูกราฟที่ 3

## กราฟที่ 3 — Histogram Live vs Spoof (density)

### ดูยังไง
- bar chart ที่แสดงการกระจายของ score แยกตาม Live (เขียว) และ Spoof (แดง)
- **แกน X**: score 0–100% แบ่งเป็น 20 ช่อง (แต่ละช่อง = 5%)
- **แกน Y**: % ของเคสในแต่ละ label ที่ score อยู่ในช่วงนี้ (ทำให้ Live และ Spoof เปรียบเทียบกันได้แม้จำนวนเคสต่างกัน)
- **เส้นฟ้าเข้ม (จุดๆ)**: threshold

### เอาไปทำอะไรต่อ

**ดูว่าระบบแยก Live vs Spoof ได้ชัดแค่ไหน**:
- ถ้า bar เขียวกองอยู่ทางขวา (score สูง) และ bar แดงกองอยู่ทางซ้าย (score ต่ำ) = ระบบแยกได้ดี
- ถ้า bar เขียวกับแดงทับซ้อนกันมาก = ระบบแยกยาก บริเวณทับซ้อนคือ "จุดเสี่ยง" ที่ threshold จะทำให้ผิดพลาดได้ทั้งสองทาง

> **ต้อง label test_case ก่อน** ถ้ากราฟว่าง ให้ไปที่ "ตั้งค่า Live / Spoof ต่อเคส" ด้านล่างสุดของหน้า แล้วกำหนดว่าแต่ละ test_case เป็น Live หรือ Spoof

## กราฟที่ 4 — FAR / FRR curve + EER

### ดูยังไง
- 2 เส้นแสดง % ของการ "ผิดพลาด" ที่ทุก threshold

| เส้น | สี | ความหมาย |
|---|---|---|
| FRR | แดง | "คนจริงโดน reject" — False Reject Rate |
| FAR | น้ำเงิน | "ปลอมเล็ดลอดผ่าน" — False Accept Rate |
| EER | จุดเหลือง | จุดที่ FRR = FAR พอดี |
| T | ฟ้าเข้ม (จุดๆ) | threshold ปัจจุบัน พร้อมค่า FRR และ FAR ตอนนี้ |

- **แกน X**: threshold 0–100%
- **แกน Y**: % ที่ผิดพลาด

ทิศทางของสองเส้น **ตรงกันข้ามกันเสมอ**:
- threshold ต่ำ → FAR สูง (ปลอมเยอะผ่านได้), FRR ต่ำ (คนจริงผ่านง่าย)
- threshold สูง → FAR ต่ำ (ปลอมผ่านยาก), FRR สูง (คนจริงโดน reject เยอะ)

### เอาไปทำอะไรต่อ

ตอบคำถาม **"threshold ไหนที่ balance ระหว่างความปลอดภัยกับ UX?"**

- **งานที่ต้องการความปลอดภัยสูง** (เช่น banking, ยืนยันตัวตนทางการเงิน): เน้นลด FAR → เลือก threshold สูงขึ้น แม้ FRR จะสูงด้วย
- **งานที่ UX สำคัญ** (เช่น log in ทั่วไป): เน้นลด FRR → เลือก threshold ต่ำลง

**EER** (จุดเหลือง) = จุดที่ FAR เท่ากับ FRR พอดี = "threshold ที่สมดุลที่สุดทางสถิติ" เป็นจุดเริ่มต้นที่ดีก่อนจะ bias ไปทางไหน

> **ต้อง label test_case ก่อน** เพื่อให้กราฟรู้ว่าเคสไหนเป็น Live เคสไหนเป็น Spoof

## กราฟที่ 5 — Box Plot ต่อเคสทดสอบ

### ดูยังไง
แต่ละ test_case มี "กล่อง" 1 ใบ ภายในกล่องประกอบด้วย:

```
           ┬   ← max
           │   (whisker บน)
    ┌──────┤
    │      │   ← Q3 (75% ของเคสอยู่ใต้นี้)
    │──────│   ← median (ส้ม) = ค่ากลาง
    │  •   │   ← mean (จุดเหลือง) = ค่าเฉลี่ย
    └──────┘   ← Q1 (25% ของเคสอยู่ใต้นี้)
           │   (whisker ล่าง)
           ┴   ← min
```

- **เส้นฟ้าเข้มแนวนอน (จุดๆ)**: threshold ปัจจุบัน

### เอาไปทำอะไรต่อ

**เปรียบเทียบ test_case ว่าใครทำคะแนนได้แตกต่างกันยังไง**:

- **กล่องสูง** (Q3-Q1 มาก) = score กระจายออก ไม่ค่อย consistent
- **กล่องเตี้ย** (Q3-Q1 น้อย) = score คงที่สม่ำเสมอ
- **กล่องอยู่เหนือ threshold ทั้งกล่อง** = เคสนี้ผ่านเกือบทุกครั้ง
- **กล่องอยู่ใต้ threshold ทั้งกล่อง** = เคสนี้ไม่ผ่านเกือบทุกครั้ง
- **threshold ผ่ากลางกล่อง** = เคสนี้ borderline — ขึ้นอยู่กับ threshold มาก ต้องพิจารณาพิเศษ

**Analogy**: นึกถึงคะแนนสอบของนักเรียนแต่ละห้อง — median คือคะแนนกลาง, กล่อง = ช่วงคะแนนของนักเรียน "ทั่วไป", whisker = คะแนนสูงสุดและต่ำสุด

## กราฟที่ 6 — Confusion Matrix + KPI ต่อเคส

### ดูยังไง
ตาราง 2×2 นับว่าที่ threshold ปัจจุบัน ระบบตอบถูก/ผิดกี่เคส:

|  | **ระบบบอกว่าผ่าน** | **ระบบบอกว่าไม่ผ่าน** |
|---|---|---|
| **ความจริงเป็น Live** | ✅ **TP** (ถูก: คนจริง ผ่าน) | ❌ **FN** (ผิด: คนจริง โดน reject) |
| **ความจริงเป็น Spoof** | ❌ **FP** (ผิด: ปลอม ผ่านเข้ามา) | ✅ **TN** (ถูก: ปลอม ถูก block) |

- ช่องสีเขียว = "ถูก", ช่องสีแดง/ส้ม = "ผิด"
- ตัวเลขใหญ่ = จำนวนเคส, ตัวเลขเล็ก = % จากทั้งหมด

**ตารางด้านล่าง** = สถิติต่อเคส (n, mean, median, p10, p90, pass%) — p10/p90 คือ percentile ที่ 10% และ 90%

### เอาไปทำอะไรต่อ

**ดูว่าระบบผิดแบบไหน และกี่เคส** ที่ threshold นี้:

- **FN สูง** = คนจริงโดน reject เยอะ → UX แย่ → อาจต้อง threshold ต่ำลง หรือปรับ model
- **FP สูง** = ปลอมเล็ดลอดผ่านเยอะ → ปลอดภัยน้อย → อาจต้อง threshold สูงขึ้น หรือเพิ่ม check อื่น
- **TP/(TP+FP)** = precision — "ที่บอกว่าผ่านแล้วถูกจริงกี่ %"
- **TP/(TP+FN)** = recall — "จาก Live ทั้งหมด จับได้กี่ %"

> **ต้อง label test_case ก่อน** เพื่อให้ตารางรู้ว่าเคสไหนเป็น Live เคสไหนเป็น Spoof

## Glossary

| คำ | ความหมายเต็ม |
|---|---|
| **face_score** | คะแนน 0–1 (0–100%) ที่ ML model ให้กับเฟรมว่าเป็นใบหน้าจริงแค่ไหน |
| **threshold** | เกณฑ์ผ่าน/ไม่ผ่าน — score ≥ threshold = ผ่าน |
| **mean (μ)** | ค่าเฉลี่ย: ผลบวกของทุก score หารด้วยจำนวนเคส |
| **median** | ค่ากลาง: เรียง score แล้วเอาตัวที่อยู่กึ่งกลาง (ไม่ถูก outlier ดึง) |
| **percentile** | ตำแหน่งในกลุ่ม เช่น p10 = score ที่ต่ำกว่า 10% ของเคสทั้งหมด |
| **Q1 / Q3** | Q1 = percentile ที่ 25%, Q3 = percentile ที่ 75% (ขอบล่าง/บนของกล่อง box plot) |
| **density** | ความหนาแน่นของข้อมูล — แกน Y ที่ทำให้พื้นที่ใต้กราฟรวมได้ 100% |
| **Live** | เคสที่ตั้งว่าเป็น "คนจริง" |
| **Spoof** | เคสที่ตั้งว่าเป็น "การหลอก" (รูป, วิดีโอ, หน้ากาก) |
| **FAR** | False Accept Rate — % ของเคส Spoof ที่ผ่านได้ (ปลอมเล็ดลอด) |
| **FRR** | False Reject Rate — % ของเคส Live ที่ถูก reject (คนจริงโดน block) |
| **EER** | Equal Error Rate — threshold ที่ FAR = FRR (จุดสมดุล) |
| **TP** | True Positive — Live + ผ่าน (ถูก) |
| **FN** | False Negative — Live + ไม่ผ่าน (ผิด) |
| **FP** | False Positive — Spoof + ผ่าน (ผิด) |
| **TN** | True Negative — Spoof + ไม่ผ่าน (ถูก) |
| **precision** | TP / (TP + FP) — จากที่บอกว่าผ่าน ถูกจริงกี่ % |
| **recall** | TP / (TP + FN) — จาก Live ทั้งหมด จับได้กี่ % |

## คำถามที่พบบ่อย (FAQ)

**Q: เข้าหน้า Analytics แล้ว spinner หมุนไม่หยุด กราฟไม่ขึ้น**
A: ตรวจสอบ 2 อย่าง: (1) มีอินเทอร์เน็ต? (2) รันแอปด้วย `--dart-define-from-file=.env.json` ไหม? ถ้ารันแบบ plain `flutter run` โดยไม่มี flag นี้ แอปจะไม่รู้ Supabase URL

**Q: กราฟ Histogram Live vs Spoof และ FAR/FRR ว่างเปล่า**
A: ยังไม่ได้ label test_case → เลื่อนลงไปที่ "ตั้งค่า Live / Spoof ต่อเคส" ล่างสุด กด chip แต่ละเคสเพื่อเลือกว่าเป็น Live หรือ Spoof

**Q: Bell curve ระฆังแคบมาก อ่าน label ไม่ออก**
A: pinch zoom เพื่อขยาย, double-tap เพื่อ reset กลับมุมมองปกติ

**Q: เปลี่ยน threshold แล้วกราฟไม่อัปเดต**
A: กราฟทุกตัวอัปเดต realtime ตาม slider — ถ้าไม่อัปเดต ลอง pull-to-refresh หรือ hot reload แอป

**Q: ทำไม mean กับ median ถึงอยู่คนละตำแหน่ง**
A: เป็นเรื่องปกติ ถ้ามีเคสที่ score สูงมากหรือต่ำมากผิดปกติ (outlier) จะดึง mean ออกไปจาก median — median บอก "กลางแบบไม่ถูกดึง" ได้ดีกว่า

**Q: ควรใช้กราฟไหนเพื่อเลือก threshold?**
A: แนะนำ 3 ขั้นตอน: (1) ดู **Pass-rate curve** เพื่อดูผลกระทบต่อ UX, (2) ดู **FAR/FRR curve** เพื่อเลือก balance ความปลอดภัย, (3) ยืนยันด้วย **Confusion matrix** ว่าเคสที่ผิดพลาดยอมรับได้ไหม
